1 · Overview

One authenticated dashboard that gathers every Claude agent instance you own — local desktop, remote computers, ephemeral sandboxes — into a single Kanban board.

Myra Agents is a Kanban board that launches CLI coding agents. Today's feature/backend-refactor branch already turned the backend into a standalone Node server and made the client connect to several servers at once (client-side fan-out): the browser/Tauri client holds N direct HTTP/WS connections and merges their boards into one view.

That works when every server is directly reachable. It breaks the moment you add a machine behind NAT, a corporate firewall, or an ephemeral sandbox — none of those can accept an inbound connection from your browser. This document describes the next step: a centralized, authenticated relay that those instances dial out to, so they appear on your board no matter where they run.

The one-line version
A stateless relay — one Cloudflare Durable Object per user — that authenticates you, tracks which of your instances are online, and forwards frames between your dashboard and your instances. It stores no boards. Each instance remains the source of truth for its own data.

The rest of this doc walks the design top to bottom: the problem, the chosen shape, every component, the wire protocol, auth, the network reality (NAT and corporate firewalls), how it stays cheap at idle, how it scales to thousands of users, the optional peer-to-peer future, and how it maps onto the existing codebase.

2 · Goal & constraints

What the user wants

  • One login, one board. Authenticate once; see all my agents merged.
  • Any location. Local desktop, a remote build box, a throwaway cloud sandbox — all first-class.
  • Behind NAT / firewalls. Remote and corporate machines that can't accept inbound connections must still join.

Engineering constraints

  • Reuse the branch. The Node server (@myra/server), its dispatch, store, and event bus should survive untouched as the per-instance backend.
  • Keep aggregation. The client's GlobalId = ${connId}::${entityId} namespacing and fan-out must keep working.
  • Scale to thousands of users without a heavy stateful tier to operate.
  • Don't centralize data. No multi-tenant board database to secure, back up, and leak.

3 · Why not direct fan-out

The current model asks the dashboard to reach each server. That's the wrong direction for the network.

Firewalls and NAT block inbound connections and allow outbound ones. A browser cannot open a connection to a laptop sitting behind a corporate firewall or a CGNAT home router — there's no reachable address and no open port. Tunnels (tailscale/cloudflared) can paper over this per machine, but that's N things to set up and there's still no shared identity.

Fig 1 — Direct fan-out (left) needs each instance reachable; the relay (right) flips the arrows so instances dial out.

The fix is to invert the direction: instances dial out to a relay over 443, which looks like ordinary HTTPS and sails through firewalls. The dashboard then talks to the relay, not to each machine. One reachable thing (the relay) instead of N.

Fan-out doesn't get deleted — it stays the LAN / self-host path. The relay is an additional, additive way in.

4 · Architecture at a glance

A stateless edge worker as the front door, one Durable Object per user as the relay, instances dialing out, and a direct shortcut for the local desktop.

Fig 2 — Full topology. Blue = RPC/WS through the hub · green = NAT-friendly outbound dial · amber = direct local shortcut · pink dashed = optional phase-2 P2P.

Reading it: your dashboard (browser or desktop) authenticates through a stateless Cloudflare Worker, which routes the connection to your Durable Object. Your instances each dial out to that same object. The object forwards commands down and events up. The desktop app additionally talks to its local sidecar directly, bypassing the hub entirely for the machine it's running on.

5 · The dumb-hub invariant

The hub authenticates, tracks presence, and forwards frames. Nothing else. It stores no boards.

This is the load-bearing decision of the whole design, so it gets its own section.

Dumb hub chosenSmart hub rejected
Holds no card data. Each instance owns its board.json and is the source of truth. Owns a cloud database of everyone's cards.
Stateless → scales horizontally for free; nothing to back up; no cross-tenant data to leak. Stateful, multi-tenant DB → isolation, backups, scaling, and a fat data-at-rest liability.
Tiny durable state: enrolled-instance list + credential references, per user. All boards, schedules, settings, run history for all users.
Why it matters
Because the hub holds no application data, its database load scales with user count, not activity, and a hub compromise leaks routing metadata — not everyone's work. Keep this invariant even under pressure to "just cache the board in the hub."

6 · Components

Cloudflare Worker — front door phase 1

A stateless edge function. On each incoming WebSocket upgrade it: (1) verifies the session JWT, (2) derives userId, (3) forwards the upgrade to env.USER_HUB.idFromName(userId) — that user's Durable Object. It holds no state and makes no routing decisions beyond "which user." Horizontally trivial: it's just a function on Cloudflare's edge.

UserHub Durable Object — the relay phase 1

A Durable Object is a single, globally-addressable, single-threaded object with its own small transactional storage. Cloudflare guarantees every connection addressed to a given id lands on the same object instance, wherever it physically lives. We allocate one per user. That single fact removes an entire distributed-systems tier (see §12). Inside it:

FieldWhat
instances: Map<id,WS>The user's online instance reverse-sockets, keyed by instanceId. Used to route an RPC to the right machine.
dashboards: Set<WS>The user's currently open dashboard sockets. Events are fanned out to all of them, tagged with instanceId.
storage (KV)Small durable facts: enrolled instances, credential references. Survives eviction. Never boards.
hibernationSockets accepted via state.acceptWebSocket() so the object can be evicted from memory while sockets stay open. See §10.

Instance connector phase 1

A small new module inside @myra/server. Behind a MYRA_HUB_URL env var, it dials out to the hub over a persistent WebSocket, registers, and serves RPC/events from that reverse channel by calling the same dispatch the HTTP route already uses (dispatchData → dispatchOs → dispatchAgent). An instance is therefore just today's server + this connector. The server itself is unchanged.

Dashboards

The same Next.js frontend, in two homes. In a browser it opens one WebSocket to the hub and sees every instance merged. In the Tauri desktop app it does the same for remote instances, but reaches its own local sidecar directly over 127.0.0.1 — lower latency, no hub round-trip, and it keeps working with no internet.

TURN / STUN phase 2 only

Not used in phase 1. They belong to the optional peer-to-peer mesh (§13).

7 · Wire protocol

Two channels meet at the hub: the instance reverse tunnel and the dashboard channel. They share the same {ok,data} envelope the existing HTTP transport already parses.

Instance ↔ hub (reverse tunnel)

The instance opens wss://hub/agent/connect with Authorization: Bearer <instance-credential>. After auth it's a persistent duplex stream of JSON frames:

DirectionFrameMeaning
instance → hub{ type:"hello", instanceId, label, capabilities }Register on connect / reconnect.
hub → instance{ type:"rpc", id, cmd, args }A command forwarded from a dashboard.
instance → hub{ type:"rpc-result", id, ok, data | error }The reply, correlated by id.
instance → hub{ type:"event", event, payload }Unsolicited push (log line, result, schedule change).
both{ type:"ping" } / { type:"pong" }Heartbeat → drives presence and keeps middleboxes from idle-killing the socket.

Dashboard ↔ hub

  • GET /api/instances → the user's online instances [{instanceId, label, status, capabilities}].
  • POST /api/i/:instanceId/rpc/:cmd (body = args) → hub forwards to that instance, awaits its rpc-result, returns the same {ok,data} envelope.
  • wss://hub/api/events → one socket; the hub multiplexes events from all the user's instances, each frame tagged with instanceId.
The payoff of symmetry
Dashboard→hub RPC returns the identical {ok,data} shape as a direct server today (http.ts). So the new hubTransport is a thin variant of the existing httpTransport — the URLs carry an instanceId and one shared WebSocket demuxes by it. Roughly one new file on the client.

8 · Auth & enrollment

Users log in once and get a session token. Machines enroll once with a pairing code and get a long-lived, revocable, per-instance credential.

User session

OIDC or email magic-link → the hub issues a short-lived session JWT (plus a refresh token). The dashboard sends it as a Bearer token on the WS upgrade and on each REST call. The Worker verifies it and derives userId — the only routing key it needs.

Instance enrollment (pairing)

Fig 4 — Pairing: a short-lived code minted by the dashboard is exchanged on the machine for a durable per-instance credential.
  1. In the dashboard (already authenticated) you click Add instance. The hub mints a one-time pairing code with a short TTL, bound to your userId.
  2. On the target machine you run the server with MYRA_HUB_URL set and myra enroll <code>.
  3. The hub validates the code and returns a long-lived instance credential (a per-instance JWT or device cert) bound to (userId, instanceId). The connector stores it and uses it to dial the tunnel.
  4. You can revoke any instance from the dashboard; its credential is invalidated and its tunnel dropped.

Tenant isolation

Every hub route filters by the session's userId, and the registry is the per-user Durable Object — so one user's connections are physically separate from another's. The per-instance server stays single-tenant; the hub is the only multi-tenant boundary, and it holds no data to leak across it.

9 · NAT & corporate networks

Outbound WebSocket over 443 is exactly what survives strict firewalls — it's why the design dials out.

Corporate firewalls block inbound and allow outbound, especially HTTPS on 443. The connector's dial-out to wss://hub looks like ordinary HTTPS to a well-known host (Cloudflare's edge), with no inbound port, no public IP, and no NAT config on the machine. That's the best-case path for a locked-down network.

Where it can still fail — and the fix

ObstacleWhat happensMitigation
TLS-inspecting proxy (DPI/MITM)Network decrypts all TLS via a corporate CA.Use the system trust store (the corporate root CA is already there). Do not pin certificates.
Forced egress proxyDirect outbound blocked; an authenticated proxy is required.Honor HTTPS_PROXY + proxy auth; do the WS upgrade via HTTP CONNECT.
Idle-killing middleboxLong-lived idle sockets get dropped.App-level ping/pong heartbeat; reconnect with jittered backoff.
Allowlist-only egressOutbound permitted only to approved domains.No architecture solves this — IT must allowlist the hub domain. A policy exception, not a code change.
The mesh is no better here
Corporate NAT is usually symmetric, so phase-2 P2P hole-punching fails and WebRTC falls back to TURN — itself an outbound connection to a relay on 443. The reverse-WS hub is the most firewall-friendly path; the mesh buys latency when direct works, never reachability in a locked-down office.

10 · Hibernation & resources

The cost of this design is holding tens of thousands of mostly-idle sockets open. Hibernation makes idle ≈ free.

You can't just close an idle instance's socket — a NAT'd machine that's disconnected can't be reached to wake it. The trick is to keep the socket open at the platform's edge while freeing the application object in memory.

Fig 5 — Active vs hibernated. Sockets stay open at the edge; the object is evicted from memory and rehydrated by the next frame.

With Cloudflare's WebSocket Hibernation API (state.acceptWebSocket()), an idle user's Durable Object is evicted from memory entirely — ~0 RAM, ~0 compute, ~0 cost — while its sockets remain connected. An incoming frame rehydrates the object, which rebuilds its in-memory maps from storage + live reconnects. This is the main reason to target Durable Objects rather than a hand-run node pool, where ~30–50KB of kernel buffer per idle socket is unavoidable.

Billing follows activity, not connections
Hibernation means you pay for messages processed, not sockets held. That makes §11 (cutting idle chatter) a direct cost lever, not just a bandwidth nicety.

11 · Adaptive log cadence

Stream live only when a human is watching. Everything else batches or stays on disk.

Line-by-line streaming is only worth its frames when someone has a run open. Agents are bursty and sparse — launch, watch for a few minutes, idle for hours — so most runs don't need live frames at all.

SituationWhat the hub carries
Interactive run, card modal openLive, line-by-line (as today).
Scheduled / headless runNothing per line. Buffer to the instance's run-log file; push a coalesced progress every few seconds, or just started/done. Full log fetched on demand via get_run_log.
Nobody watchingOnly the terminal agent-result-changed. The log already lives on the instance's disk — the stream was only ever a convenience.

Because every pushed frame wakes the hibernating Durable Object (§10), trimming idle chatter directly reduces RAM and cost — not just network.

12 · Scaling to thousands

It works at thousands of users — and the DO-per-user model deletes the distributed tier you'd otherwise have to build.

The load model

Say 5,000 users × ~2–4 instances = ~10k–20k always-open sockets, plus a few thousand dashboards → order ~15k–25k concurrent WebSockets. The defining cost is holding them, not data: throughput is low except live log streaming, and §11 trims that.

The trap a naive hub hits

With a hand-run node pool you need ≥2 nodes for availability before you ever need them for load. The moment you cross one node, a dashboard's RPC must reach the specific node holding that instance's socket — forcing a shared presence store (Redis), an inter-node bus (pub/sub), and reconnect-storm handling. That's the real cost behind "operate the hub."

Fig 6 — DO-per-user: all of a user's connections colocate in one object, so there is nothing to route between. No Redis, no bus, no sticky sessions.

Durable Objects make that whole tier vanish: every connection for a user is colocated in one object by construction. Cloudflare runs millions of tiny DOs; per-user routing and isolation are the platform's job, not yours.

What still needs care

  • Whale tenants. A single user with thousands of instances bottlenecks their one single-threaded DO. Shard such a user across multiple DOs by instance-group. Normal users never feel it.
  • Reconnect storms. A deploy drops every tunnel; they all redial at once. Jittered backoff on the connector; connection draining on rollout.
  • Sandbox churn. Ephemeral instances connect/disconnect rapidly. Keep presence ephemeral (TTL), no DB row per session.
  • Message cap. DO frames cap at ~1MB — fine for log lines and RPC; fetch large artifacts on demand, never ship them over the channel.
Trade-off: vendor lock-in
This is a Cloudflare-shaped design. Keep the wire protocol (§7) transport-agnostic so a self-hosted NATS/Redis hub remains buildable from the same connector — that's your portability hedge.

13 · Phase 2 — E2E P2P mesh optional

Move the data plane peer-to-peer so the hub carries only signaling — lower latency and the hub goes blind to your data.

In phase 1 all traffic flows through the hub (trust: hub-trusted). The optional mesh keeps the hub for identity, discovery, and signaling only; actual command/log traffic goes directly dashboard↔instance over WebRTC, encrypted end-to-end.

Fig 7 — Mesh: the hub brokers the handshake (signaling); data goes direct via STUN, or relays through TURN when direct fails.
STUNCheap. Tells each peer its public ip:port so they can hole-punch a direct path. No data passes through it.
TURNThe fallback relay when hole-punching fails (symmetric NAT — common for sandboxes and corporate networks). It forwards the packets. You run and pay for it (bandwidth).
E2E (DTLS)Encryption is end-to-end regardless of path. Even relayed through TURN, the relay sees only ciphertext. TURN costs bandwidth, never privacy.

Why it's deferred: it adds a full WebRTC/ICE stack on both ends, it's hard to debug, and symmetric-NAT cases fall back to TURN anyway — so it doesn't improve reachability over the phase-1 hub, only latency and privacy when direct connections succeed. Build it as a data-plane layer over the same control plane, not as a starting point.

14 · Client integration

The existing aggregation layer barely changes: one new transport, and each instance becomes a connection.

  • connId = instanceId. Each instance the hub reports becomes its own client Connection. The existing GlobalId = ${connId}::${entityId} namespacing (aggregate/global-id.ts) is untouched — ids never collide and "which instance owns this" stays answerable.
  • New hubTransport. A thin variant of http.ts: RPC URLs carry the instanceId; one shared WebSocket per hub demuxes events by instanceId. Slots into the existing Transport interface, so ConnectionManager.invokeAll / invokeOne / listenAll (manager.ts) keep working.
  • Mixed transports per board. ConnectionManager already pairs each connection with its own transport (buildTransport). The desktop keeps its local connection on tauriTransport/127.0.0.1 (direct) and adds hub-instance connections on hubTransport — local-direct + hub-for-the-rest falls out for free.
  • One WS, many connections. Where fan-out opens N WebSockets today (listenAll), hub-instances share one hub socket; the transport demuxes frames by instanceId to the right connection's listeners.

16 · Risks & mitigations

RiskMitigation
Blast radius. The hub holds live reverse tunnels into machines that can spawn shell commands; a compromise approaches RCE on every connected box.Per-instance credentials (never a shared secret), capability scoping in hello, short-TTL signed tokens, revocation from the dashboard, and a roadmap to E2E (§13) so the hub routes ciphertext.
Privacy. Phase-1 hub-trusted means the operator can see traffic and prompts.Be explicit about it; offer the self-hosted gateway and the phase-2 mesh for users who need the hub blind.
Vendor lock-in. DO-shaped design.Transport-agnostic wire protocol → a self-hosted NATS/Redis hub buildable from the same connector.
Reconnect storms / whale tenants / sandbox churn.Jittered backoff, connection draining, per-user DO sharding, ephemeral TTL presence (§12).
Always-on dependency. The hub is a 24/7 service.Stateless edge + DO removes the node pool; the self-hosted gateway is the fallback for users who'd rather own uptime.

Simpler fallbacks

  • Tunnel-per-instance (tailscale/cloudflared): gives each remote box a reachable URL so today's fan-out works unchanged, zero new services. Ship the feature now, defer the hub.
  • Self-hosted gateway: the same protocol, but the relay is a binary the user runs on one box they already expose. Write the routing once, never operate it.
